# 内容哲学 — 专业程序员版

> **何时阅读此文件：** 在阶段 3（编写模块 HTML）期间。这些原则指导每个内容决策。

这些原则是将优秀技术课程与普通教程区分开来的关键。

### 信息密度优先

专业程序员阅读速度快，理解能力强。课程应该信息密度高，减少"废话"。

**文字原则：**
- 每个段落有明确信息增量。如果没有新信息，删掉它。
- 直接使用技术术语，无需解释基础概念。
- **图表和动画优先于代码贴片。** 能用架构图、数据流动画、时序图表达的，就用可视化。
- **代码仅作为补充。** 仅在图表无法替代时才展示代码，且每片段控制在 10-15 行。
- 60%+ 的屏幕应该是图表、动画或交互元素，25% 是简洁的文字描述，15% 是精选代码。

**将描述转换为可视化（优先级排序）：**
- "组件 A 调用组件 B" → **数据流动画**（首选）或**时序图**
- "数据从 X 流向 Y" → **数据流动画**
- "系统架构" → **交互式架构图**（点击组件显示描述）
- "请求处理流程" → **流程图** + **步骤卡片**
- "状态变化" → **状态图动画**
- "这个函数的核心逻辑" → **精简代码片段（10-15 行）+ 设计意图注释**（仅在可视化不足以表达时使用）

**避免的做法：**
- 不要贴超过 15 行的代码块 — 如果需要展示更多代码，拆分为多个小片段，每个片段聚焦一个概念
- 不要连续放置两个代码块而中间没有可视化元素
- 不要用代码来展示可以用图表表达的关系和流程

### 代码分析块 — 精选且聚焦

面向专业程序员的代码分析，重点不是"这段代码做什么"（他们能看懂代码），而是"为什么要这样设计"。**一个课程中只需要 2-3 个代码分析块**，每个聚焦最核心的设计决策。

**代码选取原则：**
- 每片段 **不超过 10-15 行** — 选取最能体现设计意图的核心代码
- 优先选取：接口定义、核心数据结构、关键算法的核心逻辑
- 不选取：配置代码、样板代码、工具函数、导入语句

**结构：**
- 左侧：精选的核心代码（保持格式，允许水平滚动）
- 右侧：设计意图注释（不是逐行翻译）

**示例：**


### 架构图 — 课程的核心视觉元素

架构图是课程中**最重要的元素**。每个课程必须包含至少 2 个架构图。使用交互式架构图（点击组件显示描述）。

**架构图类型：**
- **组件图** — 展示模块边界和依赖关系
- **层次图** — 展示分层架构（表现层、业务层、数据层）
- **部署图** — 展示服务拓扑和数据流

**设计原则：**
- 清晰的边界和箭头方向
- 使用颜色区分不同类型的组件
- 标注关键数据流路径
- 包含图例
- **每个组件可点击**，显示职责描述和关键接口

### 数据流动画 — 追踪请求生命周期

数据流动画是仅次于架构图的第二重要元素。每个课程至少 2 个。展示请求从入口到响应的完整旅程。

**内容：**
- 入口点（路由、控制器）
- 中间件链
- 业务逻辑层
- 数据访问层
- 外部服务调用
- 响应返回路径

**动画化：** 使用 `data-steps` 属性创建分步动画，让学习者逐步追踪数据流。

### 技术方案描述 — 简洁文字替代大段代码

对于技术方案和实现细节，优先使用**简洁的文字描述 + 可视化**，而不是贴大段代码。

**描述模式：**
- **方案概述** — 用 2-3 句话说明技术方案的核心思路
- **关键决策** — 用提示框（callout）标注为什么选择这种方案
- **实现要点** — 用编号步骤卡片（step-cards）列出关键实现步骤
- **核心代码** — 仅在上述方式无法充分表达时，展示 10-15 行最核心的代码

**示例（好的方式）：**
> 认证系统采用 JWT + Refresh Token 双 token 方案。Access Token 有效期 15 分钟，存储在内存中；Refresh Token 有效期 7 天，存储在 HttpOnly Cookie 中。Token 验证通过中间件统一处理，失败后自动尝试刷新。
>
> [数据流动画：用户请求 → 中间件验证 → Token 过期 → 刷新流程 → 重新请求]

**示例（差的方式）：**
> [贴 50 行 auth middleware 代码]
> [贴 30 行 token refresh 代码]
> [贴 20 行 cookie 设置代码]

### 测验（可选功能）

**重要：测验默认不包含。** 仅当用户明确要求"增加测验"或"需要测验题"时才添加。

测验应该测试学习者能否**应用**知识解决新问题，而不是回忆事实。

**测验类型（按价值排序）：**

1. **设计决策题** — "如果要添加 X 功能，你会选择哪种方式？为什么？"
   ```
   题目：团队决定添加一个批量导出功能，预期数据量可能达到 100 万条。
   你会选择哪种实现方式？

   A. 同步导出，直接在内存中处理后返回文件
   B. 异步任务队列，处理完成后发送邮件通知
   C. 流式处理，边读边写，不占用太多内存

   正确答案：C 或 B（取决于具体需求）
   解释：同步导出会超时；异步适合大规模但需要用户等待通知；
   流式处理内存效率最高，但需要实现复杂的流控逻辑...
   ```

2. **架构修改题** — "如果要将单体拆分为微服务，你会如何划分边界？"

3. **问题诊断题** — "用户报告 X 功能变慢，根据你学的架构，可能是什么原因？"

4. **代码定位题** — "如果要修改 Y 行为，你需要修改哪些文件？"

**不测验的内容：**
- 文件名记忆
- 语法细节
- 可以通过 Ctrl+F 找到的事实

### 设计笔记提示框

使用提示框标注重要的设计决策和权衡：

```html
<div class="callout callout-info">
  <div class="callout-title">设计笔记</div>
  <p>这里使用策略模式而不是 if-else，是为了支持未来添加新的支付方式而不修改核心逻辑。这是开闭原则的应用。</p>
</div>
```

**类型：**
- **设计决策** — "为什么这样设计"
- **权衡说明** — "选择了 A 而不是 B，因为..."
- **陷阱警告** — "注意：这里容易出错"
- **扩展点** — "如果要添加 X，可以在这里扩展"

### 技术术语处理

专业程序员了解基础术语，但可能不熟悉：
- 项目特定的术语和缩写
- 特定框架/库的概念
- 公司内部约定

**处理方式：**
- 项目特定术语：首次出现时用词汇表提示
- 通用技术术语：直接使用，不解释
- 框架特定概念：简要说明与项目的关联

### 模块间的技术深度递进

模块之间的深度应该递进，可视化复杂度也应递进：

1. **模块 1-2**：架构层面 — 交互式架构图、组件关系图、层次图，用文字描述整体设计思路
2. **模块 3-4**：实现层面 — 数据流动画、时序图、精选核心代码片段（2-3 个），用步骤卡片描述关键流程
3. **模块 5-6**：进阶层面 — 状态图、场景对比图，用提示框讨论扩展策略和技术债务

不要在早期模块深入代码细节，也不要在后期模块重复架构概念。

### 实用性导向

每个模块都应该回答一个实用问题：
- "我在哪里添加新功能？"
- "我如何定位 bug？"
- "我如何理解这个错误？"
- "我如何扩展这个系统？"

如果模块不能帮助学习者解决实际问题，重新设计它。
