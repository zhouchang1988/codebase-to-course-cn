# 内容哲学 — 产品经理版

> **何时阅读此文件：** 在阶段 2.5（编写模块简报）和阶段 3（编写模块 HTML）期间。这些原则指导每个内容决策 —— 展示什么、如何解释以及用什么方式呈现。

这些原则是让产品经理真正看懂系统的关键。

### 核心原则：业务语言，零代码

产品经理需要理解"系统做了什么"和"业务逻辑怎么流转"，不需要知道"代码怎么写"。

**绝对禁止：**
- 任何代码片段（不论长短）
- 代码翻译块
- 技术实现细节（算法名称、数据结构、设计模式术语）
- 编程术语的词汇表提示
- 测验题

**语言规范：**
- 用业务语言描述系统行为："用户提交订单后，系统自动校验库存"而非"调用 checkInventory() 方法"
- 用"模块"、"服务"、"功能"等产品经理熟悉的词汇
- 避免"类"、"函数"、"接口"、"中间件"等编程术语
- 可以提及技术组件的名字（如"数据库"、"缓存"、"消息队列"），但只解释它在业务中的角色

### 展示，不要讲述 —— 极致视觉化

产品经理习惯看流程图和架构图。课程应该更像产品文档而不是技术文档。

**文字限制：**
- 每个文字块最多 **3-4 个句子**。如果你在写第五句，停下来把它转换为视觉元素。
- 每个屏幕必须 **至少 60% 是视觉内容**（流程图、架构图、卡片、模块关系图）。

**将文字转换为视觉元素：**
- 步骤描述 → **流程图**（带清晰的开始/结束/分支）
- "模块 A 和模块 B 协作" → **系统架构图**（带箭头和标注）
- "这个功能包含 X、Y、Z" → **功能模块卡片**（图标 + 一句话描述）
- "用户先做 A，再做 B" → **用户旅程图**（带步骤编号）
- "数据从 X 流到 Y" → **数据流转动画**
- 列表描述 → **带图标的卡片网格**
- 条件逻辑 → **决策树/分支流程图**

**视觉呼吸空间：**
- 在元素之间使用充足的间距
- 在全宽视觉元素和窄文字块之间交替，创造节奏
- 每个模块应该至少有一个"英雄视觉" —— 一个主导屏幕的图表或交互元素

### 业务流程图 — 课程的核心

业务流程图是产品版课程中**最重要的元素**。每个模块至少一个。

**流程图类型：**
- **用户操作流程** — 用户从开始到完成某个目标的完整路径
- **业务逻辑流程** — 系统内部业务规则的执行顺序和分支
- **审批/状态流程** — 数据或实体的状态变迁
- **异常处理流程** — 出错时系统的应对策略

**设计原则：**
- 用业务语言标注每个节点（"校验用户身份"而非"auth middleware"）
- 分支条件用业务条件表述（"库存充足？"而非"inventory > 0"）
- 标注关键的输入和输出
- 使用颜色区分正常路径和异常路径

### 系统架构概览 — 让PM看到全景

用简化的架构图展示系统的"大图"。不需要技术细节，只需要模块边界和职责。

**展示内容：**
- 各功能模块的名称和一句话职责
- 模块之间的调用/依赖关系（用箭头和简短标注）
- 外部系统的集成点
- 数据存储的位置和用途

**不展示：**
- 具体的技术实现（框架名、库名）
- 代码层面的依赖关系
- 部署细节

### 用户旅程图 — 端到端的产品体验

从用户的视角描述使用系统的完整流程。

**结构：**
- 明确的起点（用户意图）和终点（目标达成）
- 每一步标注：用户做了什么 → 系统做了什么 → 用户看到什么
- 标注关键的决策点和分支路径
- 标注可能出错的地方和系统的应对

### 功能模块卡片 — 职责一目了然

每个模块用一张卡片展示其核心信息。

**卡片包含：**
- 模块名称（业务名称，不是代码文件名）
- 一句话职责描述
- 核心功能列表（3-5 条）
- 与其他模块的关系（谁给它数据、它给谁数据）

### 数据流转 — 讲清楚"数据去哪了"

产品经理最关心的问题之一：数据从哪来、怎么处理、到哪去。

**展示方式：**
- 用动画展示数据在系统中的流动路径
- 每个节点标注"对数据做了什么"（而非"用什么技术做的"）
- 标注数据格式的变化（如"用户填的表单" → "存储的订单记录"）
- 标注数据持久化的位置

### 每个屏幕一个概念

不要堆砌信息。模块内的每个屏幕讲清楚一件事。

**好的屏幕主题：**
- "订单创建的完整流程"
- "支付模块和库存模块如何配合"
- "系统如何处理退款请求"

**差的屏幕主题：**
- "后端架构和所有API" — 太大了，拆分
- "技术栈介绍" — 产品经理不需要这个

### 业务规则可视化

系统中的业务规则是产品经理最需要理解的部分。

**展示方式：**
- 用决策树展示条件逻辑（"如果 VIP 用户 → 免运费；如果新用户 → 首单折扣"）
- 用状态图展示实体的生命周期（订单：待支付 → 已支付 → 已发货 → 已完成）
- 用表格对比不同场景的处理方式

### 模块间内容递进

模块之间应该从宏观到微观递进：

1. **模块 1-2**：全景层 — 系统做什么、有哪些大模块、核心价值
2. **模块 3-4**：流程层 — 核心业务流程、用户旅程、数据流转
3. **模块 5-6**：细节层 — 模块协作细节、边界情况、扩展方向

### 实用性导向

每个模块都应该回答产品经理关心的问题：
- "这个系统的核心业务逻辑是什么？"
- "用户操作后系统内部发生了什么？"
- "各模块之间是怎么配合的？"
- "数据是怎么流转和存储的？"
- "如果要加新功能，会影响哪些模块？"
- "系统的边界在哪里？和哪些外部系统有交互？"

如果模块不能帮助产品经理理解业务逻辑或做出产品决策，重新设计它。
